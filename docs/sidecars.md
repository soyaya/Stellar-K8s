# Custom Sidecars in StellarNode

The `sidecars` field on `StellarNodeSpec` lets you inject additional containers into the pod generated by the operator. Sidecars share the pod's network namespace and can mount the same volumes as the main `stellar-node` container.

## Shared Volumes

Two volumes are always available to sidecars:

| Volume name | Type | Contents |
|-------------|------|----------|
| `data` | PersistentVolumeClaim | Stellar Core ledger data, history archives, and logs |
| `config` | ConfigMap | Generated `stellar-core.cfg` / Horizon / Soroban config |

Mount them read-only in your sidecar to avoid interfering with the main process.

## Example: Custom Log Processing with Fluent Bit

```yaml
apiVersion: stellar.org/v1alpha1
kind: StellarNode
metadata:
  name: validator-with-logging
  namespace: stellar-system
spec:
  nodeType: Validator
  network: Mainnet
  version: "v21.0.0"
  storage:
    storageClass: standard
    size: 100Gi
  validatorConfig:
    seedSecretSource:
      localRef:
        name: validator-seed
  sidecars:
    - name: log-forwarder
      image: fluent/fluent-bit:3.0
      resources:
        requests:
          cpu: "50m"
          memory: "64Mi"
        limits:
          cpu: "100m"
          memory: "128Mi"
      volumeMounts:
        # Mount the data volume read-only to tail stellar-core logs
        - name: data
          mountPath: /stellar-data
          readOnly: true
      env:
        - name: LOG_LEVEL
          value: "info"
      # Fluent Bit config can be provided via a separate ConfigMap mounted here
      # or baked into a custom image
```

Stellar Core writes logs to stdout/stderr (captured by Kubernetes) and optionally to files under the data directory. Point Fluent Bit (or any log shipper) at `/stellar-data` to tail file-based logs, or use a shared `emptyDir` volume if you need to pass logs between containers via a named pipe.

## Example: Prometheus Metrics Proxy

```yaml
  sidecars:
    - name: metrics-proxy
      image: prom/node-exporter:latest
      ports:
        - containerPort: 9100
          name: metrics
      resources:
        requests:
          cpu: "25m"
          memory: "32Mi"
        limits:
          cpu: "50m"
          memory: "64Mi"
```

## Example: Custom Proxy / Splunk Forwarder

```yaml
  sidecars:
    - name: splunk-forwarder
      image: splunk/universalforwarder:9.2
      env:
        - name: SPLUNK_START_ARGS
          value: "--accept-license"
        - name: SPLUNK_FORWARD_SERVER
          value: "splunk-indexer.internal:9997"
      volumeMounts:
        - name: data
          mountPath: /stellar-data
          readOnly: true
      resources:
        requests:
          cpu: "100m"
          memory: "128Mi"
        limits:
          cpu: "200m"
          memory: "256Mi"
```

## Notes

- Sidecars are appended after the main `stellar-node` container in the pod spec. The main container is always index 0.
- The operator does not validate sidecar names for conflicts — ensure your sidecar names do not clash with operator-managed containers (`stellar-node`, `nat-traversal`, `cloudhsm-client`, etc.).
- Resource requests/limits on sidecars count toward the pod's total resource budget. Size them appropriately.
- See `examples/sidecar-example.yaml` for a complete working manifest.
