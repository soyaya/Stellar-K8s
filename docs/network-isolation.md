# Network Isolation Architecture

Cost-conscious teams often share Kubernetes clusters across environments. This document describes how the Stellar-K8s operator guarantees that a **Testnet node can never accidentally connect to a Mainnet peer or share a database**, even on a shared cluster.

---

## Problem Statement

Stellar nodes are network-aware: a validator configured for the Testnet will only form quorum with other Testnet validators, and a Horizon instance ingests ledger data from a specific network passphrase. However, at the infrastructure level, Kubernetes pods are just processes — without explicit controls, nothing prevents:

- A Testnet pod from opening a TCP connection to a Mainnet pod's peer port (11625).
- A Horizon instance from being pointed at the wrong Stellar Core URL.
- An operator bug from creating a Mainnet resource in a Testnet namespace.

The consequences range from wasted compute (failed consensus) to data corruption (mixed ledger history in a shared database).

---

## Isolation Architecture

Isolation is enforced at **two independent layers**. Both must be defeated for a cross-network connection to succeed.

```
┌─────────────────────────────────────────────────────────────────────┐
│  Layer 1: Control-Plane (Reconciler)                                │
│                                                                     │
│  check_network_safety() runs at the start of every reconcile loop. │
│  If any StellarNode in the same namespace is configured for a       │
│  different network, reconciliation is blocked with SK8S-021 and     │
│  a Kubernetes Warning event is emitted.                             │
│                                                                     │
│  This prevents the operator from ever creating cross-network        │
│  resources, even if the data-plane policies are misconfigured.      │
└─────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│  Layer 2: Data-Plane (NetworkPolicy)                                │
│                                                                     │
│  Two sets of NetworkPolicies enforce isolation at the kernel level: │
│                                                                     │
│  A. Per-node policies (generated by the reconciler)                 │
│     • Ingress: allow only from same-network namespaces + monitoring │
│     • Egress:  allow only to same-network namespaces + DNS          │
│     • Policy type: Ingress + Egress (deny-by-default for both)      │
│                                                                     │
│  B. Namespace-level policies (deployed by Helm)                     │
│     • deny-non-mainnet-ingress / deny-non-mainnet-egress            │
│     • deny-non-testnet-ingress / deny-non-testnet-egress            │
│     • Applied to ALL pods in the namespace, not just Stellar pods   │
└─────────────────────────────────────────────────────────────────────┘
```

## Namespace Labelling Convention

Every namespace that hosts Stellar nodes **must** carry the label:

```
stellar.org/network: <network>
```

Where `<network>` is one of:

| Value       | Stellar Network                                    |
|-------------|----------------------------------------------------|
| `mainnet`   | Public Global Stellar Network ; September 2015     |
| `testnet`   | Test SDF Network ; September 2015                  |
| `futurenet` | Test SDF Future Network ; October 2022             |
| `custom-*`  | Custom network (hash-derived, DNS-safe)            |

### Applying the label

```bash
kubectl label namespace stellar-mainnet stellar.org/network=mainnet
kubectl label namespace stellar-testnet stellar.org/network=testnet
```

Or via Helm (see [Helm Configuration](#helm-configuration) below).

## Layer 1: Control-Plane Safety Check

**Source:** `src/controller/network_isolation.rs`

The function `check_network_safety(client, node)` is called at the start of every `apply_stellar_node` reconcile, before any Kubernetes resources are created or modified.

### What it checks

1. **Peer node scan** — Lists all `StellarNode` resources in the same namespace. If any peer is configured for a different `spec.network`, reconciliation is blocked.

2. **Namespace label verification** — Reads the namespace's `stellar.org/network` label. If the label is present and mismatches the node's network, a warning is logged. (Missing label is tolerated during initial setup; the data-plane policies are the hard barrier.)

### Error code

`SK8S-021` — emitted as a Kubernetes `Warning` event on the `StellarNode` resource and returned as a reconciler error (triggering exponential backoff).

### Example event

```
Warning  NetworkSafetyViolation  stellar-operator
[SK8S-021] Network safety violation in namespace 'stellar-shared':
node 'mainnet-validator' is configured for network 'mainnet' but
node 'testnet-validator' in the same namespace is configured for
network 'testnet'. Mainnet and Testnet nodes MUST reside in separate
namespaces to prevent accidental cross-network peering or database sharing.
```

## Layer 2: Data-Plane NetworkPolicies

### A. Per-node policies (reconciler-generated)

**Source:** `src/controller/resources.rs` — `build_network_policy()`

When `spec.networkPolicy.enabled: true` on a `StellarNode`, the reconciler creates a `NetworkPolicy` resource in the node's namespace. The policy:

**Ingress rules** (allow):
- Traffic from namespaces labelled `stellar.org/network=<same-network>`
- Traffic from explicitly listed namespaces (`spec.networkPolicy.allowNamespaces`)
- Traffic from pods matching `spec.networkPolicy.allowPodSelector`
- Traffic from CIDRs in `spec.networkPolicy.allowCidrs`
- Prometheus scraping from the monitoring namespace (port 9090)
- Validator peer traffic from same-network pods (port 11625)

**Egress rules** (allow):
- Traffic to namespaces labelled `stellar.org/network=<same-network>`
- DNS (UDP/TCP 53 to kube-dns in kube-system)
- Intra-namespace traffic (e.g. Horizon → Stellar Core in the same namespace)

**Policy types:** `Ingress` + `Egress` — both directions are deny-by-default.

### B. Namespace-level policies (Helm-deployed)

**Source:** `charts/stellar-operator/templates/network-isolation.yaml`

These policies apply to **all pods** in the namespace, not just Stellar pods. They act as a backstop if a per-node policy is accidentally deleted.

For each Mainnet namespace:
- `deny-non-mainnet-ingress` — drops ingress from any namespace not labelled `stellar.org/network=mainnet`
- `deny-non-mainnet-egress` — drops egress to any namespace not labelled `stellar.org/network=mainnet`

For each Testnet namespace:
- `deny-non-testnet-ingress` — drops ingress from any namespace not labelled `stellar.org/network=testnet`
- `deny-non-testnet-egress` — drops egress to any namespace not labelled `stellar.org/network=testnet`

Both sets allow:
- kube-system (DNS)
- The operator namespace (health probes, event publishing)
- The monitoring namespace (Prometheus scraping, if `allowMonitoringNamespace: true`)

## RBAC Scoping

**Source:** `charts/stellar-operator/templates/rbac.yaml`

The operator's RBAC is structured to minimise cross-namespace access:

| Permission | Scope | Rationale |
|------------|-------|-----------|
| `stellarnodes`, `stellarbenchmarks`, `benchmarkreports` | Namespace (or cluster if `watchNamespace` unset) | Core CRD management |
| `pods`, `services`, `configmaps`, `pvcs`, `deployments`, `statefulsets` | Namespace (or cluster if `watchNamespace` unset) | Workload management |
| `networkpolicies` | Namespace (or cluster if `watchNamespace` unset) | Per-node isolation policies |
| `secrets` | Read-only, namespace-scoped | Seed secret references |
| `namespaces` | Read-only, cluster-scoped (separate ClusterRole) | Network safety label check |
| `leases` | Namespace-scoped | Leader election |

The `namespaces` read permission is intentionally isolated in a separate `ClusterRole` (`<release>-namespace-reader`) so teams running with `watchNamespace` set can audit and grant only this narrow cluster-level permission.

### Recommended deployment for shared clusters

```yaml
# values.yaml for a shared cluster
operator:
  watchNamespace: "stellar-mainnet"  # or "stellar-testnet"

networkIsolation:
  enabled: true
  mainnetNamespaces:
    - stellar-mainnet
  testnetNamespaces:
    - stellar-testnet
  allowMonitoringNamespace: true
  monitoringNamespace: monitoring
```

With `watchNamespace` set, the operator's main Role is namespace-scoped. The only cluster-level permission is the read-only `namespaces` ClusterRole, which cannot be used to modify any resource.

## Helm Configuration Reference

```yaml
networkIsolation:
  # Master switch. Set to true to deploy namespace-level isolation policies.
  enabled: false

  # When true, the Helm chart patches the release namespace with the
  # stellar.org/network label specified in releaseNamespaceNetwork.
  labelReleaseNamespace: false
  releaseNamespaceNetwork: ""   # e.g. "mainnet" or "testnet"

  # Namespaces hosting Mainnet nodes.
  mainnetNamespaces: []
  # Example:
  # mainnetNamespaces:
  #   - stellar-mainnet
  #   - stellar-mainnet-horizon

  # Namespaces hosting Testnet nodes.
  testnetNamespaces: []

  # Allow Prometheus to scrape metrics across the network boundary.
  allowMonitoringNamespace: true
  monitoringNamespace: "monitoring"
```

## StellarNode Configuration Reference

Per-node network policy is controlled via `spec.networkPolicy`:

```yaml
apiVersion: stellar.org/v1alpha1
kind: StellarNode
metadata:
  name: my-validator
  namespace: stellar-mainnet
spec:
  nodeType: Validator
  network: Mainnet
  networkPolicy:
    enabled: true                        # required to activate per-node policy
    allowNamespaces:                     # additional namespaces allowed in
      - stellar-mainnet-horizon
    allowMetricsScrape: true             # allow Prometheus scraping
    metricsNamespace: monitoring
```

## Operational Runbook

### Diagnosing SK8S-021

```bash
# Check events on the affected StellarNode
kubectl describe stellarnode <name> -n <namespace> | grep -A5 NetworkSafetyViolation

# List all StellarNodes in the namespace and their networks
kubectl get stellarnodes -n <namespace> -o custom-columns=\
  NAME:.metadata.name,NETWORK:.spec.network,TYPE:.spec.nodeType

# Check the namespace label
kubectl get namespace <namespace> --show-labels
```

**Resolution:** Move the conflicting node to a namespace labelled for its network, or relabel the namespace if all nodes in it share the same network.

### Verifying NetworkPolicy enforcement

```bash
# List per-node NetworkPolicies
kubectl get networkpolicies -n <namespace> -l app.kubernetes.io/managed-by=stellar-operator

# List namespace-level isolation policies
kubectl get networkpolicies -n <namespace> -l stellar.org/policy-type=network-isolation

# Test connectivity (should be blocked)
kubectl run test-pod --image=busybox --rm -it --restart=Never \
  -n stellar-testnet -- wget -T3 http://mainnet-horizon.stellar-mainnet.svc.cluster.local:8000
# Expected: connection timeout
```

### Adding a new network namespace

1. Create the namespace and apply the label:
   ```bash
   kubectl create namespace stellar-mainnet-2
   kubectl label namespace stellar-mainnet-2 stellar.org/network=mainnet
   ```

2. Add it to the Helm values and upgrade:
   ```yaml
   networkIsolation:
     mainnetNamespaces:
       - stellar-mainnet
       - stellar-mainnet-2
   ```
   ```bash
   helm upgrade stellar-operator ./charts/stellar-operator -f values.yaml
   ```

3. Deploy StellarNodes into the new namespace with `spec.networkPolicy.enabled: true`.

## Security Considerations

- **CNI dependency:** NetworkPolicies are enforced by the CNI plugin. Ensure your cluster uses a CNI that supports NetworkPolicy (Calico, Cilium, Weave Net, etc.). Flannel does **not** enforce NetworkPolicy by default.
- **Namespace label integrity:** The `stellar.org/network` label on a namespace is the trust anchor for the data-plane policies. Restrict who can label namespaces using Kubernetes RBAC or OPA/Gatekeeper.
- **Operator namespace:** The operator namespace itself is allowed through the isolation policies (for health probes and event publishing). Ensure the operator namespace is not shared with untrusted workloads.
- **Monitoring namespace:** If `allowMonitoringNamespace: true`, ensure the monitoring namespace is not shared with untrusted workloads that could exploit the allowed ingress path.
